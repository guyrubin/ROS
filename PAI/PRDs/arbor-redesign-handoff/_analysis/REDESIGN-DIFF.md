# Arbor Redesign — Prototype vs Live App Diff

**Date:** 2026-06-23  
**Author:** arbor-pm  
**Source documents:**  
- Prototype: `PAI/PRDs/arbor-redesign-handoff/_analysis/PROTOTYPE-CATALOGUE.md`  
- Live baseline: `PAI/PRDs/arbor-redesign-handoff/_analysis/LIVE-APP-INVENTORY.md`  
**Critical principle:** The live app is RICHER than the prototype in several dimensions. The redesign is a UX/IA/design-system upgrade, NOT a feature expansion. Implementing the prototype naively would regress the live app. Every Bucket-A ticket is a shell upgrade wired to the live backend.

---

## BUCKET A — NET-NEW (in prototype, not in live app)

These items exist in the prototype but are absent from the live app. Each is a candidate ticket. Each note names what live capability it RESKINS or REORGANIZES — the builder wires the new shell to the existing rich backend, no rebuild from scratch.

### A-1. Sidebar + Topbar Desktop Information Architecture
**What the prototype adds:** A persistent left sidebar (6-view primary nav) + a topbar chrome layer (global search, notifications bell, multi-kid switcher) as the desktop navigation paradigm. The live app uses `Shell.tsx` with collapsible tabs.  
**Reskins/reorganizes:** The existing 34-tab navigation in `Shell.tsx` and `navigation.ts`. No backend change — routing stays; the chrome changes.  
**Gate:** safe (pure frontend chrome; no data surface). Wave 1.

### A-2. Global Search
**What the prototype adds:** A topbar global search field (prototype web-only feature). No equivalent exists in the live app.  
**Reskins/reorganizes:** Indexes over existing content — milestones, activities in `playbank/content.ts`, coach topics, journey names in `heroJourneys.ts`. Search is a discovery layer on top of the existing data, not a new store.  
**Gate:** safe (read-only query; no child-data egress). Wave 2.

### A-3. Notifications / Notification Bell
**What the prototype adds:** A topbar notification bell with a count indicator. The live app has no notification center UI.  
**Reskins/reorganizes:** The existing JITAI nudge engine (`lib/jitai.ts`, fixed by AP-005) and any monitoring alerts already generated by `monitoring.ts`. The bell surfaces signals already computed — it does not generate new ones.  
**Gate:** safe (displaying already-computed signals; no new data write or push consent). Wave 2.

### A-4. Multi-Kid Switcher with Add-Child in Topbar
**What the prototype adds:** A topbar child-chip ("Mia, 6") with a multi-kid switcher and an explicit Add Child affordance embedded in the topbar. The live app has a profile switcher (`ProfileSwitcher.tsx`) but it is buried in Profile, not topbar-accessible.  
**Reskins/reorganizes:** The existing multi-child management system (`ProfileContext.tsx`, `ProfileSwitcher.tsx`) and the Add Child flow already in the app. The topbar chip is a new access point, not a new capability.  
**Gate:** safe (frontend placement; existing child-data path unchanged). Wave 2.

### A-5. Kid Mode Overlay (Desktop)
**What the prototype adds:** An explicit "Kid Mode" overlay on the desktop web shell — a separate child-facing world accessible from the parent shell via a mode toggle. On mobile the live app already has a parent/kid world switch; the desktop has no equivalent.  
**Reskins/reorganizes:** The existing child-facing surfaces (HeroArcade, HeroJourneyTab, the Feelings check-in, the practice worlds) already in the live app. The overlay is a new entry mechanism on desktop, wiring to the existing child routes.  
**Gate:** safe (frontend mode toggle; no new data surface). Wave 3.

### A-6. 5-Step Onboarding Flow
**What the prototype adds:** A structured 5-step onboarding (Welcome → Child setup → Focus selection → Avatar/hero creation → Ready) with progress dots, Back/Continue/Skip controls, and re-launch capability. The live app has `OnboardingFlow.tsx` but it is a minimal stub (age slider only — AP-008 addresses the months bug).  
**Reskins/reorganizes:** The existing `OnboardingFlow.tsx`, `ProfileContext.tsx`, focus-domain selection (already a product concept), and the avatar creation pipeline (`AvatarCreator.tsx`) that already exists and is already production-deployed.  
**Gate:** gated (the "Focus selection" step at step 3 touches developmental domain labeling and could imply a screen if copy is wrong — requires arbor-clinical-psych copy pass; step 4 avatar-creation is child-data per COPPA/GDPR). Wave 3.

### A-7. Design System / Token Layer
**What the prototype adds:** A unified design token layer: accent-theme switching (Green/Teal/Blue), a coherent type scale, spacing tokens, component primitives. The live app has 344 hardcoded hex literals scattered across components (per `arbor-app-architecture.md`) with no token system.  
**Reskins/reorganizes:** Every existing UI component. This is NOT a feature — it is the foundation wave that unlocks all other visual upgrades cleanly. Owned by `arbor-design`. No backend change.  
**Gate:** safe (pure styling; no data or clinical surface). Wave 1 — must land before visual tickets.

### A-8. Avatar Fan-Out to All Surfaces (Hero Avatar Engine)
**What the prototype adds:** The Hero Avatar Engine: one shared on-device canvas module compositing the avatar across 8+ surfaces (Hero Card, comic cover + 6 panels, Academy hero, Studio stamp, reward/badge art, breathing-guide buddy, milestone celebration). The live app has an avatar generator and uses it in stories and comics (`comics` tab PARTIAL, `HeroJourneyTab` FULL) but has no shared canvas module — each surface re-implements compositing.  
**Reskins/reorganizes:** The existing avatar generation (`AvatarCreator.tsx`, Gemini vision pipeline) and story/comic render paths (`HeroJourneyTab.tsx`, `heroJourneys.ts`). The canonical build rule from the Blueprint: one shared module, add a surface by passing a new template. No new avatar data — reuses the saved avatar.  
**Gate:** safe (frontend render module; zero new child-data surface; avatar is already generated and stored locally). Wave 4.

### A-9. Day Windows — Hour-by-Hour Calm/Hard-Time Prediction
**What the prototype adds:** A parent-facing "Day Windows" dashboard showing age-based calm/hard-time prediction by hour, pattern detection (e.g. "mornings before preschool hard 5/7 days"), and a "sharpens as you log" value loop. No equivalent surface exists in the live app, though the JITAI engine (`lib/jitai.ts`) already computes rhythm predictions.  
**Reskins/reorganizes:** The existing JITAI rhythm engine and behavior log data (`BehaviorsTab.tsx`, `behaviors/PatternInsights.tsx`). Day Windows is a visualization layer over signals already computed.  
**Gate:** gated (surfaces a predictive developmental signal to the parent; requires arbor-clinical-psych copy pass to confirm non-pathologizing framing; child-data read from behavior log). Wave 4.

### A-10. Accent Theme Switching (Green/Teal/Blue)
**What the prototype adds:** A parent-facing accent color preference (3 themes). The live app has no theme switching.  
**Reskins/reorganizes:** The design token layer (Bucket A-7 / AP-043). This item is a thin extension once the token layer lands; it is a Wave 4 polish item after Wave 1 ships the token foundation.  
**Gate:** safe (pure styling preference; no data surface). Wave 4.

### A-11. Development Copilot + Learning Map in Academy
**What the prototype adds:** A "Development Copilot" pane in Academy (suggests focusing on the lowest-scoring domain, with objectives list) and a Learning Map (course progress rolled up by the same 7 domains as the Development Map). The live app has a `copilot` tab (FULL) and Academy `masterclasses` tab, but they are not surfaced as a unified "Copilot + Learning Map" view inside Academy.  
**Reskins/reorganizes:** The existing `copilot` route data and Academy course structure in `playbank/content.ts`. No new AI call needed — the copilot already generates recommendations; this is a new display location and visual join.  
**Gate:** safe (existing data, new display surface). Wave 4.

### A-12. Language Lab (Bilingual Vocabulary Tracker)
**What the prototype adds:** A "Language Lab" dashboard with bilingual He/En vocabulary counts and a balance-over-time trend chart (~620 He / ~410 En words, 62% balance). The live app has a `language` tab (FULL — multilingual phrase cards, Serve & Return) but no vocabulary count tracker or balance chart.  
**Reskins/reorganizes:** The existing `language` tab data model and i18n infrastructure (`LanguageContext.tsx`, `i18n.ts`). Adds a count/trend visualization on top of existing phrase-card data.  
**Gate:** gated (vocabulary counts are derived from child-logged phrase data; clinical copy must not imply a bilingual development screen — requires arbor-clinical-slp framing pass). Wave 4.

**KEEP LIVE — do not downgrade.** In every dimension below, the live app ships more than the prototype shows. Implementing the prototype number naively would be a hard regression.

Prototype 6 games are a subset. Wire the redesigned Playbank shell to all 14 live worlds. |
| Hero journeys / stories | **10 hero journeys** (4 packs, 8-beat spine, choices, metrics) | 4 stories (David & Goliath, Noah's Ark, Dragon of Responsibility, Lion Who Was Afraid) | KEEP LIVE — 10 journeys. The 4 prototype stories are a subset of the 10. |

| Pricing / entitlement tiers | **Free / Plus / Family** (RevenueCat wired) | Free + Plus only (no Family in prototype data model) | KEEP LIVE — Family tier. Do not strip Family tier from the live billing stack. |
